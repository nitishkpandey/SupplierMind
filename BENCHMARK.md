# SupplierBench-25

The benchmark protocol for the three-paradigm comparison, and how to
reproduce it end-to-end from a clean checkout.

## Dataset

- **Corpus**: synthetic suppliers generated by `backend/data/generate_dataset.py`
  with `random.seed(42)` — same output every run. Two sizes: the 100-supplier
  curated set and the 10,000-supplier scale set
  (`backend/scripts/generate_synthetic_suppliers.py` + `bulk_ingest_synthetic.py`).
- **Queries**: 25 procurement queries in `backend/data/queries_benchmark.json`,
  each with ground-truth supplier ids and a difficulty tier:
  - **easy** (1-2 constraints), **medium** (3-4), **hard** (5-6).
- **Cert prevalence** differs between corpora by design — measured in
  `results/diagnostics/cert_prevalence.csv`; interpretation notes in
  `Documents/csr_at_scale_defence.md` (the 10K corpus is cert-sparser, i.e.
  the harder one).

## Systems under test

| System | Description |
|---|---|
| P1 | Single-prompt LLM (no corpus) — scored by strict normalised-name matching against corpus names |
| P2 | Minimal RAG (top-10 retrieve, one prompt, pick 5) |
| P3 | SupplierMind multi-agent pipeline |
| Baseline A | Keyword SQL search (no LLM) |
| Baseline B | Manual-process simulation (no LLM) |

All LLM-bearing systems run on the same provider stack (GPT-4o-mini primary,
Groq fallback — see `backend/app/core/llm.py`).

## Metrics

- **P@5** — precision of the top-5 against ground-truth ids.
- **MRR** — reciprocal rank of the first relevant result.
- **CSR** — constraint satisfaction rate; for P3 from the compliance matrix,
  for others from direct field comparison.
- **Latency** — wall-clock ms per query.
- **Cost** — estimated USD per query (provider cost logging).
- Bootstrap 95% CIs (1,000 resamples) on headline metrics for the final run.

## Model versions

| Component | Model | Notes |
|---|---|---|
| Primary LLM | `gpt-4o-mini` (OpenAI) | benchmark runs |
| Fallback LLM | `llama-3.1-8b-instant` (Groq) | retryable-failure fallback; Groq-era archive in `results/groq_baseline_v0/` |
| Embeddings | `voyage-3-lite` (512-dim) | both retrieval and query memory |

All prompts are versioned in the repository (no out-of-band prompt edits):
agent prompts in `backend/app/agents/`, baseline prompts in
`backend/experiments/`.

## Reproduction (clean checkout)

```bash
# 0. Prerequisites: Docker Desktop, Python 3.11, uv, API keys in backend/.env
docker compose up -d

cd backend
uv sync                                     # pinned deps (uv.lock committed)
uv run alembic upgrade head                 # schema
uv run python data/generate_dataset.py      # corpus + benchmark (seed 42)
uv run python scripts/ingest_suppliers.py   # embed + index into Milvus

# Full three-paradigm benchmark (GPT-4o-mini; ~30-60 min, paced)
uv run python scripts/run_evaluation.py --paradigms

# Outputs:
#   backend/data/evaluation_results.json    per-query + aggregated metrics
#   backend/data/thesis_report.json         thesis-ready report
```

Archive the run by copying the outputs into `results/run_YYYYMMDD/` before
re-running anything — locked results are never overwritten in place.

One-command wrapper (does all of the above):

```powershell
./scripts/reproduce_benchmark.ps1
```

## Honest-reporting protocol

Negative results stay. Numbers are not re-run to look better. If P3
underperforms P2 anywhere, that goes into the results unchanged — the point
of the three-paradigm framing is to learn where each approach wins and loses.
